# Contributing to Cockroach Chat

Thanks for being here. This is a mesh messenger meant to work when the network doesn't, and it is
built by a small number of people — outside help genuinely moves it.

**Found a security vulnerability? Stop and read [`SECURITY.md`](SECURITY.md).** Report privately,
never in a public issue.

## What helps most right now

1. **Security review.** The code is unaudited and much of it was AI-authored (see
   [How this was built](README.md#how-this-was-built)). Careful review of `crates/meshcore` —
   the wire codec, relay logic, and rate limiting — is the highest-value contribution available.
2. **Hardware reports.** We've validated on a Galaxy S23 and a OnePlus. BLE behaviour varies wildly
   across OEMs and Android versions. If you have two phones and ten minutes, file a
   [hardware report](https://github.com/xdadwal/cockroach-chat/issues/new?template=hardware-report.yml) —
   including "it didn't work," which is more useful than silence.
3. **The performance backlog.** [`docs/PERFORMANCE.md`](docs/PERFORMANCE.md) is a written,
   prioritised list — several items are self-contained and make good first contributions.

**Not looking for iOS work right now.** The project is Android-only and iOS is deferred — please
don't start a CoreBluetooth port expecting it to be merged. See [`ROADMAP.md`](ROADMAP.md).

## Project invariants

These are non-negotiable. A PR that breaks one will be asked to change, no matter how good it is
otherwise — each exists because a predecessor project died on it.

1. **Vetted crypto only.** Use `snow`, `ed25519-dalek`, `x25519-dalek`, `chacha20poly1305`, and the
   RustCrypto family. **Never hand-roll a primitive.** Every parser gets size and decompression
   caps and must be fuzzable — a zip bomb killed Bridgefy's entire mesh, and a signature-parsing
   overflow broke bitchat.
2. **The core is sans-IO.** No threads, no async runtime, and no clock syscalls inside `meshcore`.
   Time, transport, and storage are injected. This is what makes hundreds of virtual nodes
   deterministically replayable in the simulator — breaking it silently destroys our ability to
   test the mesh at all. (It's also why the core uses `BTreeMap`/`BTreeSet`: `HashMap` iteration
   order was breaking reproducibility.)
3. **Don't drift from [`docs/protocol.md`](docs/protocol.md).** It is normative. If you must change
   the wire format, bump `PROTOCOL_VERSION` **and** regenerate the golden vectors in the same
   commit: `REGEN_VECTORS=1 cargo test -p meshcore --test vectors`. CI diffs the committed vectors
   against freshly generated bytes and will fail otherwise.
4. **Respect the scope.** The target is local clusters of ~50–500 people in physical proximity, not
   city-scale chat. BLE physics doesn't allow the latter and we don't pretend otherwise. The v2
   deferral list in [`docs/IMPLEMENTATION_PLAN.md`](docs/IMPLEMENTATION_PLAN.md) is enforced.

## Setup

### Rust core — works on any desktop, no phone needed

This is the whole protocol, and most contributions never need an Android device.

```bash
cargo test --workspace                                    # unit tests + scenarios
cargo clippy --workspace --all-targets -- -D warnings
cargo fmt --all -- --check

cargo run -p sim -- --nodes 200 --scenario broadcast      # deterministic mesh simulator
```

Simulator scenarios: `broadcast`, `partition`, `storm`, `flooder`, `relay`, `two`.

### Fuzzing the parsers

The parsers see fully untrusted bytes off the air, and parser bugs are what broke our
predecessors — so they're fuzzed. Targets live in `crates/meshcore/fuzz` (nightly + `cargo-fuzz`,
kept out of the workspace so `cargo build --workspace` still works on stable):

```bash
cargo install cargo-fuzz
cd crates/meshcore/fuzz
cargo +nightly fuzz run wire_decode -- -max_total_time=60
```

Targets: `wire_decode` (packet decoding plus an encode/decode round-trip), `frag_reassemble`
(stateful reassembly with adversarial frame sequences), `decompress` (the zip-bomb caps). CI runs
60 s per target on every PR and 20 minutes per target nightly.

**If you add or change a parser, add a fuzz target for it.**

### Android app

Needs Rust stable with the Android targets, an Android SDK + NDK, and a JDK 17+.

```bash
rustup target add aarch64-linux-android armv7-linux-androideabi x86_64-linux-android

# Cross-compile the core and regenerate the Kotlin UniFFI bindings.
# The NDK is auto-detected from a standard SDK location; override with ANDROID_NDK_HOME.
./scripts/build-android-lib.sh

# Faster while iterating — emulator-only ABI:
ABIS=x86_64 ./scripts/build-android-lib.sh

cd android && ./gradlew installDebug     # JAVA_HOME must point at a JDK 17+
```

macOS and Linux hosts are both supported. If `JAVA_HOME` isn't set, Android Studio's bundled JDK
works: `/Applications/Android Studio.app/Contents/jbr/Contents/Home` on macOS.

The `.so` libraries, JNA, and generated bindings are packaged into the APK automatically. Note that
`jniLibs/` and the generated `uniffi/` bindings are gitignored — they're build outputs, so run the
script rather than looking for them in git.

### Testing without two phones

Real BLE needs **physical devices** — emulators have no Bluetooth radio, so the emulator build
falls back to an in-app loopback transport. You can exercise the UI and the core on an emulator,
but any claim about mesh behaviour needs either the simulator or real hardware.

## Making a change

1. **Open an issue first** for anything non-trivial, so you don't build something we'd reject on an
   invariant.
2. **Write the test first.** The core is test-driven; new protocol behaviour needs a unit test, and
   new mesh behaviour usually wants a simulator scenario.
3. **Run the checks locally** — `cargo test --workspace`, `cargo clippy --workspace --all-targets --
   -D warnings`, `cargo fmt --all -- --check`. CI runs exactly these and treats warnings as errors.
4. **Verify on hardware** if you touched BLE or UI, and say so in the PR. `FLAG_SECURE` blocks
   screenshots, so describe what you saw rather than trying to capture it.
5. **Keep commits focused** and write messages that explain *why*, not just what.

### Review

- Most changes need **one maintainer approval**.
- Changes to crypto, the wire format, or key handling need **two** — see
  [`GOVERNANCE.md`](GOVERNANCE.md). This is slower on purpose.

### Style

Match the surrounding code. Rust is `rustfmt`-formatted; Kotlin follows the existing Compose
patterns in `android/app/src/main/java/chat/cockroach/ui/`. Comments should explain reasoning, not
restate the code — the existing codebase is a reasonable guide to the density we're after.

## Honesty rules

This is a tool people might rely on in genuinely dangerous situations, so:

- **Never claim something works without running it.** Paste the command and its output.
- **Never make the UI imply more safety than exists.** "Never fake trust, never fake connectivity"
  is the first field principle, and it applies to code as much as to design.
- If you're unsure whether something is secure, **say so in the PR**. Flagged uncertainty is
  valuable; quiet uncertainty is how people get hurt.
